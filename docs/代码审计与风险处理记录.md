## 问题 1：`semantic_proxy` 距离被高维语义向量主导

原实现把 `z_s` 和低维 proxy 直接拼接后逐维标准化，再用欧氏距离做 KNN。由于 `z_s` 通常有数百到上千维，而 proxy 只有少量维度，即使每一维都被标准化，高维块的总体距离贡献仍然会自然更大。这样会导致 `semantic_proxy` 名义上同时使用语义和复杂度代理变量，实际近似退化成 semantic-only 检索。

处理方式：

- 将条件距离改成分块标准化。
- 语义块和 proxy 块分别用 memory bank 统计量标准化。
- 每个块再按 `weight / sqrt(dim)` 缩放，默认 `semantic_weight=1.0`、`proxy_weight=1.0`。
- 在输出 JSON 中记录 `condition_distance` 和每个结果的 `condition_blocks`，方便复现实验设置。

影响：

- `semantic_proxy` 的含义更明确：默认让 semantic block 和 proxy block 在距离中有接近的块级贡献。
- 后续可以通过 `--semantic-weight` 与 `--proxy-weight` 做权重消融，而不是让维度差异隐式决定结果。

## 问题 2：bank 与 eval 重叠时存在自邻居泄漏

如果用 `--eval-split train` 或 `--eval-split all` 做诊断，真实样本可能同时出现在 memory bank 和 eval set。KNN 会把样本自己检索为最近邻，局部 prior 的能量会被人为压低，尤其会污染训练集 sanity check。

处理方式：

- 评估脚本现在检测 bank/eval 是否有重叠。
- 默认排除自邻居。
- 只有显式传入 `--allow-self-neighbor` 时才保留旧行为，用于复现或调试。
- 输出 JSON 中记录 `self_neighbor_policy`。

建议：

- 正式报告仍应优先使用 `bank_split=train`、`eval_split=val/test` 的非重叠设置。
- 如果必须在 train 上做 sanity check，应保留默认排除自邻居策略。

## 问题 3：proxy 审计不应使用 val/test 来决定 proxy 选择

原审计脚本直接在整个 cache 上计算 proxy 与 label、generator、operation、split 的相关性。如果根据这个结果选择 proxy，相当于把验证集或测试集分布信息提前用于设计条件变量，容易形成隐性泄漏。

处理方式：

- 新增 `--audit-split`，默认 `train`。
- label、generator、group、operation 相关风险只在 audit split 上计算。
- `all_split_eta2` 仍然保留，但只作为分布漂移警告，不作为 proxy 选择依据。
- 输出 JSON 中记录 `audit_split`、`audit_count` 和 `split_counts`。

建议：

- proxy 选择、剔除、权重设定只看 train audit。
- val/test 上的 proxy 审计只用于解释失败模式，不用于反向调参。

## 问题 4：`z_c_proxy` 不是原图复杂度真值

当前 structural proxy 是基于模型输入张量计算的，也就是 resize、center crop、normalize 后再反归一化到 `[0, 1]` 的图像。它适合定义为“模型输入尺度上的操作性代理变量”，但不能表述为原始图像复杂度真值。

处理方式：

- feature cache 新增 `proxy_frame` 字段，例如 `preprocessed_resize_short256_crop224_imagenet_denormalized`。
- README 中明确说明 structural proxy 基于预处理图像计算。
- audit/eval 输出会带上 `proxy_frame`，旧 cache 则显示 `legacy_unknown`。

建议：

- 论文中使用“operational proxy”或“preprocessed-input proxy”这类谨慎表述。
- 如果后续要主张原图复杂度，需要另做原图尺度 proxy 缓存和压缩/resize 控制实验。

## 问题 5：JSON 输出可能包含非标准 `NaN`

部分分组指标在样本不足时会出现 `NaN`。Python 默认 `json.dump` 会把它写成非标准 JSON，后续用严格解析器、可视化工具或实验平台读取时可能失败。

处理方式：

- `save_json` 现在会递归清理 numpy 标量、数组和非有限浮点数。
- `NaN`、`Inf`、`-Inf` 会转成 `null`。
- `json.dump(..., allow_nan=False)` 强制输出标准 JSON。

影响：

- 指标缺失会以 `null` 形式显式出现，便于下游工具处理。

## 问题 6：旧版 GMM baseline 的 percentile 语义容易误读

`fit_beyond_gmm.py` 中 `--threshold-percentile` 原默认值为 `0.05`，但 `np.percentile` 的单位是百分位，因此实际是第 0.05 百分位，不是常见理解中的 5%。这会让阈值过于保守，导致操作点不可比。

处理方式：

- 默认值改为 `5.0`。
- 增加参数范围校验 `(0, 100)`。
- 输出中记录 `threshold_percentile` 和 `threshold_false_alarm_target`。

影响：

- 默认操作点现在对应约 5% calibration false alarm，更符合低误报区间评估习惯。

## 问题 7：缓存字符串与错误信息的可维护性

旧 cache 使用 object string array，需要 `allow_pickle=True` 才能读取；部分错误信息存在乱码，不利于 AutoDL 日志排查。

处理方式：

- 新 cache 的字符串数组改为 numpy unicode string。
- 评估脚本暂时保留 `allow_pickle=True`，兼容旧 cache。
- 将已发现的乱码错误信息改为英文 ASCII 文本。

建议：

- 重新生成 Phase 1 cache，以获得 `proxy_frame`、`sample_index` 和 unicode string array。
- 后续确认不再需要旧 cache 后，可以进一步去掉 pickle 兼容读取。

## 问题 8：单一类别 split 的 balanced accuracy 显示异常

在只包含 real 或只包含 fake 的诊断 split 上，balanced accuracy 本身不可定义。旧实现的阈值搜索会在所有候选阈值都得到 `NaN` 时保留初始值 `-1.0`，最终格式化成 `-100%`，容易让人误以为模型表现极差。

处理方式：

- `best_balanced_threshold` 现在会跳过非有限值。
- 如果没有任何可定义的 balanced accuracy，则返回 `NaN`。
- `compute_binary_metrics` 在这种情况下也会把 balanced-threshold 相关指标保持为缺失值。

影响：

- 单类分组的不可定义指标会在 JSON 中输出为 `null`，在日志中显示为 `nan`，不会再出现误导性的负百分比。

## 问题 9：`BeyondLowLevelAdapter` 存在未训练随机 projection

旧实现加载并冻结 low-level encoder 后，又新建了一个 `LayerNorm -> Linear -> GELU -> Linear` projection head。这个 projection head 不来自 checkpoint，也不会在 `cache_ccfled_features.py` 的特征缓存阶段训练。这样得到的 `z_f` 带有一次固定但随机的坐标变换，会削弱 forensic embedding 的可解释性和跨运行可复现性。

处理方式：

- `BeyondLowLevelAdapter` 不再创建未训练 projection head。
- 当 `out_dim` 与 checkpoint `feature_dim` 不一致时直接报错。
- 新 cache 的 `config_json` 和顶层字段记录 `lowlevel_projection="none"`。

影响：

- Phase 1 的 `z_f` 现在直接来自训练过的 low-level encoder token。
- 如果后续确实需要降维或投影，应作为显式训练模块保存到 checkpoint，而不是在提特征时临时初始化。

## 问题 10：验证阶段缺少 grad 开关

部分训练脚本通过 `model.train(train)` 切换训练/验证模式，但验证 forward 仍然会构建 autograd graph。虽然没有 optimizer step，但会增加显存和时间开销。

处理方式：

- `train_lowlevel_precomputed.py`、`train_lnp_extractor.py`、`train_probe.py` 和 `train_conditional.py` 的 `run_epoch` 已使用 `torch.set_grad_enabled(train)`。

影响：

- 验证阶段不再构建梯度图。
- 训练阶段行为保持不变。

## 问题 11：scorer fitting 需要基于显式 feature cache

意见“先缓存特征，再拟合 scorer”是合理的。否则每个 baseline 自己提特征，会导致不同脚本之间的 backbone、projection、pooling、split 和预处理细节难以对齐，Phase 1 的 global/local prior 对比也很难复现。

处理方式：

- 新增 `scripts/evaluation/fit_feature_scorer.py`。
- 统一从 `cache_ccfled_features.py` 生成的 `.npz` 读取 `z_f`、`z_s`、`z_c_proxy` 和 split/meta。
- 支持 `global_diag`、`global_gmm`、`local_semantic`、`local_proxy`、`local_semantic_proxy`。
- 所有 scorer 都输出统一 JSON，并可保存 `.joblib` scorer model。

保留策略：

- `fit_beyond_gmm.py` 暂时保留为 legacy manifest 直跑 baseline。
- 主实验和论文结果优先使用 `fit_feature_scorer.py`，避免重复提特征带来的不一致。

## 问题 12：conditional classifier 不应成为主故事默认路线

意见“retrieval condition first”是合理的。主论文假设是 `z_c` 作为检索/控制条件，而不是直接参与真假判别。如果把所有 condition 喂给 classifier，模型容易学习 semantic 或 pipeline shortcut，这和 local comparable real prior 的故事线冲突。

处理方式：

- 主路径改为 cache-based scorer：先用 `z_s/z_c_proxy` 检索可比真实锚点，再在 `z_f` 上计算偏离分数。
- README 中把 `fit_feature_scorer.py` 作为 scorer fitting 入口。
- 现有 `train_conditional.py` 保留为 legacy conditional detector baseline，不作为 Phase 1 主结果接口。

没有做的事：

- 没有把 `train_conditional.py` 硬改成 retrieval trainer。这样会把一个 legacy classifier 改成半套新方法，反而更难复现。

## 问题 13：decorrelation 应实现为 cross-cov penalty

意见“不要只写正交损失口号，而要做 cross-cov penalty”是合理的。逐样本向量正交通常不稳定，也容易受尺度影响；跨 batch/token 的 cross-covariance penalty 更接近“让两个表征块统计上少共享线性信息”的目标。

处理方式：

- 新增 `cross_covariance_penalty(a, b)`。
- `train_conditional.py` 新增 `--lambda-decor`，默认 `0.0`，用于 legacy conditional baseline 的可选消融。
- 计算 decorrelation 时 residual block 默认 `detach()`，避免为了 decorrelation 破坏 forensic residual。

建议：

- Phase 1 主实验不需要开启该项。
- 如果后续做 Phase 2/3 的表征解耦实验，再用 `--lambda-decor` 做系统消融。

## 问题 14：固定方形 resize 会改变长宽比

用户最终确定主协议采用大多数类似视觉工作的 ImageNet-style resize/crop，而不是 `Resize((224, 224))` 的方形拉伸。该选择合理：它保留原图长宽比，同时保持模型输入为固定 `224x224`。

处理方式：

- `build_image_transform(train=True)` 改为 `Resize(256) -> RandomCrop(224) -> RandomHorizontalFlip -> ToTensor -> Normalize`。
- `build_image_transform(train=False)` 改为 `Resize(256) -> CenterCrop(224) -> ToTensor -> Normalize`。
- `Resize(256)` 使用 torchvision 的整数形式，表示短边缩放到 256，不是拉伸到 `256x256`。
- `PairedManifestDataset` 使用同步 crop/flip，real/fake pair 共享同一次随机裁剪和翻转。
- `train_lowlevel_precomputed.py` 对 original 和所有 diffusion variants 使用同步 crop/flip，避免 pretext label 被随机裁剪差异污染。
- `prepare_beyond_precomputed.py` 生成 variants 时使用 eval 协议，即 `Resize(256) -> CenterCrop(224) -> ToTensor`。

影响：

- 后续重新生成的 cache 会记录 `transform_protocol="resize_short256_crop224"`。
- `z_c_proxy` 仍是模型输入 crop 上的代理变量，不是原图全局复杂度。

```bash
  --k 32 \

```

AutoDL 单张 RTX 5090 上建议先保持 `batch_size=32`，确认显存和吞吐后再尝试 `64`。

## 仍需谨慎的结论边界

- `z_c_proxy` 仍然只是代理变量，不是复杂度真值。
- 固定 K 仍然需要稳定性检查，不能只报告单个 K。
- `semantic_weight` 和 `proxy_weight` 是实验超参，应该做消融。
- 如果 proxy 与 generator、operation 或 split 强相关，不能直接声称复杂度建模有效。
- Phase 1 的主 claim 应限制在“局部可比真实先验是否优于全局真实先验”，不要提前宣称完整 CC-FLED 已经解决跨域检测。
